**Payload는 Next.js 풀스택 프레임워크입니다.** Payload Config를 작성하면 즉시 다음을 얻을 수 있습니다:

- React 서버/클라이언트 컴포넌트를 사용하는 완전한 관리자 패널 - 데이터 구조와 일치하며 자체 React 컴포넌트로 완전히 확장 가능
- 마이그레이션, 트랜잭션, 적절한 인덱싱 등을 포함한 직접 DB 액세스 및 소유권이 있는 자동 데이터베이스 스키마
- 즉각적인 REST, GraphQL, 그리고 직접적인 Node.js API
- 자체 앱에서 사용할 수 있는 인증
- 깊이 커스터마이징 가능한 접근 제어 패턴
- 크롭/초점 선택과 같은 파일 저장소 및 이미지 관리 도구
- 라이브 프리뷰 - 업데이트하는 즉시 프론트엔드에서 콘텐츠 변경사항을 실시간으로 확인
- 그 외 많은 기능들

### 즉각적인 백엔드 슈퍼파워

무엇을 만들든 Payload는 백엔드 슈퍼파워를 제공합니다. 전체 Payload 설정은 기존 Next.js 앱에 한 줄로 설치할 수 있으며, 개발 프로세스를 비약적으로 향상시키도록 설계되었습니다. Payload는 모든 현대적인 웹 앱의 가장 복잡하고 시간이 많이 걸리는 부분을 단순하게 만듭니다.

### 오픈 소스 - Vercel을 포함한 어디서나 배포 가능

MIT 라이선스를 가진 완전한 오픈 소스이며 Node.js 앱을 실행할 수 있는 곳이라면 어디서나 자체 호스팅할 수 있습니다. 또한 기존 Next.js 애플리케이션 내에서 Vercel과 같은 호스트에 서버리스로 배포할 수도 있습니다.

### 코드 우선 및 버전 관리

Payload에서는 "클릭 작업"이 없습니다 - 관리자 패널에서 클릭하여 스키마를 정의하는 것이 아닙니다. Payload에서는 모든 것이 올바른 방식으로 이루어집니다—적절한 백엔드처럼 코드 우선이고 버전 관리됩니다. 하지만 개발자가 Payload가 어떻게 작동해야 하는지 정의하면, 기술적이지 않은 사용자들도 코드를 전혀 알 필요 없이 관리자 패널을 사용하여 필요한 것을 독립적으로 관리할 수 있습니다.

### 완전히 확장 가능

기본적으로 제공되는 많은 기능에도 불구하고, 데이터베이스, 관리자 UI 또는 기타 앱의 모든 측면을 완전히 제어할 수 있습니다. Payload의 모든 부분은 현대적인 TypeScript/React로 확장 및 커스터마이징 가능하도록 설계되었습니다. 그리고 작성하는 코드를 완전히 이해할 수 있습니다.

## 사용 사례

Payload는 헤드리스 콘텐츠 관리 시스템(CMS)으로 시작했지만, 이후 커뮤니티가 단순히 페이지와 블로그 포스트를 관리하는 것을 훨씬 넘어서는 방식으로 Payload를 활용하는 것을 보았습니다. 이제는 풀스택 TypeScript 앱 프레임워크로 성장했습니다.

대기업들은 중요한 내부 도구를 구동하는 데 Payload를 사용하고, 소매업체들은 헤드리스 Shopify 없이도 전체 상점을 운영하며, 방대한 양의 디지털 자산이 Payload 내에서 저장 및 관리됩니다. 물론, 크고 작은 웹사이트들도 여전히 콘텐츠 관리를 위해 Payload를 사용합니다.

### 헤드리스 CMS

마케터들이 인용하는 대규모 웹 프로젝트의 가장 큰 장벽은 엔지니어링입니다. 반대로 엔지니어들은 그 반대라고 말합니다. 이는 수많은 CMS 옵션이 있음에도 불구하고 아직 해결되지 않은 큰 문제입니다.

Payload는 라이브 프리뷰, 리디렉션, 폼 빌더, 비주얼 편집, 정적 A/B 테스팅 등과 같은 기능으로 개발자/마케터 관계에 다시 사랑을 불어넣었습니다. 하지만 마케팅 효율성에 대한 이 모든 초점에도 불구하고, 우리는 개발자 경험을 타협하지 않습니다. 그렇게 해서 엔지니어와 마케터 모두가 그들이 만드는 제품에 자부심을 가질 수 있습니다.

웹사이트를 구축하고 있고 프론트엔드가 Next.js에 있다면, Payload는 당연한 선택입니다.

<Banner type="success">
  두 개의 완전히 별개의 관심사를 관리해야 하고, 양방향으로 거의 또는 전혀 네이티브 연결이 없는 SaaS 벤더에 가입하는 대신, 기존 Next.js 저장소에 Payload를 설치하기만 하면 즉시 완전한 CMS를 얻을 수 있습니다.
</Banner>

공식 Website 템플릿을 사용하여 CMS로서 Payload를 시작하세요:

```
npx create-payload-app@latest -t website
```

### 기업용 도구

대규모 조직이 새로운 소프트웨어 이니셔티브를 시작할 때, 처리해야 할 많은 기본 작업이 있습니다.

- ORM 또는 Ruby on Rails나 Laravel과 같은 앱 프레임워크로 데이터 레이어 구축
- 인증을 위한 SSO 제공자 구현
- 권한 부여를 위한 접근 제어 패턴 설계
- 필요한 REST 엔드포인트 개방 또는 GraphQL 쿼리/뮤테이션 구현
- 시간이 지남에 따라 변경되는 데이터베이스에 대한 마이그레이션 워크플로우 구현
- 웹훅 시스템 또는 유사한 것을 제작하여 다른 타사 솔루션과 통합

그리고 [관리자 패널](../admin/overview)이 있습니다. 대부분의 기업용 도구는 관리자 UI가 필요하며, 처음부터 구축하는 것은 새로운 기업용 도구의 가장 시간이 많이 걸리는 측면일 수 있습니다. Rails와 같은 앱 프레임워크용 기성 패키지가 있지만, 종종 커스터마이징이 너무 복잡해서 Material UI나 유사한 것을 처음부터 사용하는 것이 더 나을 수 있습니다.

그런 다음 사용할 수 있는 노코드 관리자 빌더가 있습니다. 그러나 접근 제어를 연결하고 적절한 버전 관리와 함께 데이터 레이어에 연결하는 것도 어려운 작업입니다.

바로 여기서 Payload가 빛을 발합니다. Payload는 이 모든 것을 즉시 제공하여 복잡한 내부 도구를 매우 간단하게 구축하고 시간이 지남에 따라 유지 관리할 수 있게 합니다. 작성해야 할 유일한 커스텀 코드는 커스텀 비즈니스 로직입니다. 즉, Payload는 일정을 단축하고 예산을 낮게 유지하며 엔지니어가 복잡한 백엔드/관리자 UI 배관보다는 특정 요구 사항에 집중할 수 있도록 합니다.

일반적으로 새로운 기업용 도구를 시작하기에 가장 좋은 곳은 자체 기능을 정의할 수 있는 빈 캔버스입니다:

```
npx create-payload-app@latest -t blank
```

### 헤드리스 커머스

UX를 우선시하는 회사들은 일반적으로 전통적인 커머스 벤더의 프론트엔드 제약에 부딪힙니다. 이러한 회사들은 Next.js와 같은 프론트엔드 프레임워크를 선택하여 사용자 경험을 최대한 미세 조정할 수 있습니다—전환 촉진, 경험 개인화, SEO 최적화.

하지만 헤드리스 커머스를 위해 Next.js와 같은 것을 사용할 때의 문제는 기술적이지 않은 사용자가 상점을 관리하려면 즉시 헤드리스 커머스 제품과 헤드리스 CMS를 결합해야 한다는 것입니다. 그러면 편집자들은 다른 기능을 위해 서로 다른 관리자 UI 사이를 오가야 합니다. 프론트엔드에서 이들을 원활하게 연결하는 데 필요한 코드는 지나치게 복잡해집니다.

Payload는 Stripe와 같은 모든 결제 프로세서와 통합할 수 있으며, 콘텐츠 작성 기능을 통해 상점의 모든 측면을 한 곳에서 관리할 수 있습니다.

단일 백엔드로 상점을 구축하고 결제 처리와 같은 것만 오프로드할 수 있다면, 코드는 더 간단해지고 편집 경험은 크게 간소화됩니다. 제품, 카탈로그, 페이지 콘텐츠, 미디어 등을 모두 한 곳에서 관리하세요.

### 디지털 자산 관리

Payload의 API 우선 태깅, 정렬 및 쿼리 엔진은 CMS가 일반적으로 저장하는 모든 유형의 콘텐츠에 완벽하게 적합하지만, 이러한 강력한 기반은 또한 강력한 디지털 자산 관리(DAM) 도구로도 만듭니다.

위의 이커머스 사용 사례와 유사하게, 조직이 콘텐츠에는 CMS를 사용하지만 디지털 자산에는 별도의 DAM을 사용한다면, 두 도구의 관리자는 밀접하게 관련된 작업을 위해 완전히 다른 서비스를 저글링해야 합니다. 두 개의 구독을 관리해야 하고, 두 세트의 인프라를 프로비저닝해야 하며, 두 개의 관리자 UI를 사용/학습해야 합니다.

Payload는 CMS와 DAM을 어느 쪽도 타협하지 않는 단일 도구로 통합합니다. 폴더 기반 구성, 파일 버전 관리, 대량 업로드, 미디어 접근 제어와 같은 강력한 기능을 통해 Payload는 동시에 완전한 디지털 자산 관리 플랫폼과 콘텐츠 관리 시스템으로 기능할 수 있습니다.

DAM으로서 Payload를 시작하는 방법에 대한 자세한 정보는 [여기를 클릭](https://payloadcms.com/use-cases/digital-asset-management)하세요.

## 프레임워크 선택하기

Payload는 모든 규모와 유형의 애플리케이션에 훌륭한 선택이지만, 모든 프로젝트에 적합한 선택은 아닐 수 있습니다. 다음은 Payload가 프로젝트에 적합한 선택인지 결정하는 데 도움이 되는 몇 가지 지침입니다.

### Payload가 적합할 수 있는 경우

- 데이터 소유권과 개인정보 보호가 중요하고, 다른 독점 SaaS 벤더가 데이터를 호스팅하고 소유하는 것을 허용하고 싶지 않은 경우
- CMS가 필요한 Next.js 사이트를 구축하는 경우
- SaaS API 외부에서 데이터를 재사용해야 하는 경우
- 구축하는 것이 일반적인 헤드리스 CMS를 넘어서는 커스텀 비즈니스 로직 요구 사항이 있는 경우
- Vercel과 같은 플랫폼에서 서버리스로 배포하고 싶은 경우

### Payload가 적합하지 않을 수 있는 경우

- 프로젝트를 완전히 코드로 관리할 수 있고 관리자 UI가 필요하지 않은 경우
- Webflow나 Framer와 같은 도구의 한계 내에서 맞는 웹사이트를 구축하는 경우
- 이미 완전한 데이터베이스가 있고 데이터를 어떻게든 시각화하기만 하면 되는 경우
- 미래의 어느 시점에서도 코드/데이터 소유권이 필요하지 않을 것이라고 확신하는 경우

시작할 준비가 되셨나요? 먼저 Payload에서 사용되는 몇 가지 고급 개념을 검토해 보겠습니다.
